Prompt vs RAG vs fine-tune

Work top-down — the cheapest tool that solves the problem wins. Fine-tuning is the last resort, not the first.

ApproachReach for it when…Cost / speed
PromptingBehaviour you can specify in words. Always try first.Cheapest · instant iteration
RAGModel needs facts it doesn't have — your private or current data.Cheap · no training
Fine-tuningA consistent behaviour, tone, or format prompting can't hold — and you have quality labelled data.Costly · slow · can go stale

Rule of thumb: fine-tuning teaches skills and style, not fresh facts. If the need is "know the latest" or "know my data," that's RAG.

Claude tool use, minimal

Define a tool, run the loop while stop_reason == "tool_use", feed each tool_result back with its tool_use_id.

python · claude tool loop
import anthropic

client = anthropic.Anthropic()

tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a city.",
    "input_schema": {
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
    },
}]

def run_tool(name, args):
    if name == "get_weather":
        return f"18C, light rain in {args['city']}"
    raise ValueError(name)

messages = [{"role": "user", "content": "Weather in Paris?"}]

while True:
    resp = client.messages.create(
        model="claude-sonnet-4-6",     # or claude-opus-4-8 / claude-haiku-4-5-20251001
        max_tokens=1024,
        tools=tools,
        messages=messages,
    )
    messages.append({"role": "assistant", "content": resp.content})

    if resp.stop_reason != "tool_use":
        break

    results = [
        {"type": "tool_result", "tool_use_id": b.id,
         "content": run_tool(b.name, b.input)}
        for b in resp.content if b.type == "tool_use"
    ]
    messages.append({"role": "user", "content": results})

print(next(b.text for b in resp.content if b.type == "text"))

Glossary

Gradient descent
The optimizer. Nudge each parameter in the direction that lowers the loss, repeat until it stops improving.
Backpropagation
The chain rule applied through the network to compute each weight's gradient — how much it contributed to the loss.
Parameters / weights
The numbers a model learns. "Training" means finding good values for them; a large model has billions.
Activation
A non-linear function (e.g. ReLU) applied after a layer's weighted sum. Without it, stacked layers collapse into one.
Loss
A single number measuring how wrong the model's predictions are. Training minimizes it.
Embedding
Content turned into a vector so that meaning becomes geometry — similar things sit close in the vector space.
Fine-tuning
Continuing to train a pretrained model on your own data to internalize a behaviour, style, or domain.
LoRA / PEFT
Parameter-efficient fine-tuning. Freeze the base model, train tiny add-on matrices (LoRA) — well under 1% of the weights.
RLHF
Reinforcement Learning from Human Feedback: train a reward model on human preferences, then optimize the policy against it.
DPO
Direct Preference Optimization: reach a similar result to RLHF by optimizing the policy directly on preference pairs — no separate reward model.
Constitutional AI
Anthropic's approach: the model critiques and revises its own answers against a written set of principles, so much of the feedback is AI-generated (RLAIF).
ReAct
The agent control-loop pattern: reason → act → observe → repeat, until the goal is met.
Tool use
Function calling. The model returns a structured request to run a tool; your code executes it and returns the result.
MCP
Model Context Protocol — an open standard for connecting agents to tools, data, and services. Write a connector once, plug in anywhere.
Multi-agent
Several specialized agents (roles, hand-offs, a coordinator) working toward one goal, instead of a single agent.
Eval
A test harness that runs the system on known inputs and scores the outputs — how you turn "seems reliable" into a number.
