🏛️ Level 5 Capstone

Build an agent

Prove you can build the intelligence, not just call it. Design and ship a small tool-using agent on the Claude API that completes a real multi-step task with at least one real tool — then measure how reliably it works.

The goal

Build an agent that uses at least one real tool inside a reason → act → observe loop to complete a task it couldn't do in a single reply — a research question, a data lookup, a small calculation-plus-fetch. Then write a short eval harness that runs it on several inputs and reports a success rate. You'll put the agentic half of this level to work: tool use, the agent loop, and evaluation. (The ML-foundations lessons — gradient descent, fine-tuning, alignment — are background for this build, not graded by it.)

The four build steps

① Pick a multi-step task

Choose something a single prompt can't reliably answer — it should need a real tool and more than one step. Good picks: "answer trivia by searching a reference source," "look up two facts and compare them," "fetch a page and summarize a section." Write down what "correct" looks like for a few example inputs — you'll need that for the eval.

② Define your tools

Each tool is {name, description, input_schema}. Make the description prescriptive about when to use it, and back it with a real implementation (a web/Wikipedia search, an HTTP call, a database query — not a stub).

starter · define tools
import anthropic

client = anthropic.Anthropic()

tools = [{
    "name": "search_wikipedia",
    "description": "Search Wikipedia and return the top result's summary. "
                   "Use this to look up facts about people, places, and things.",
    "input_schema": {
        "type": "object",
        "properties": {
            "query": {"type": "string", "description": "Search terms"},
        },
        "required": ["query"],
    },
}]

def run_tool(name, args):
    if name == "search_wikipedia":
        return wikipedia_summary(args["query"])   # your REAL implementation
    raise ValueError(f"unknown tool: {name}")

③ Implement the ReAct / tool loop

Call the model, and while it asks for tools (stop_reason == "tool_use"), run them and feed the results back. Always cap the loop so it can never run forever.

starter · agent loop
def run_agent(question, max_steps=6):
    messages = [{"role": "user", "content": question}]

    for _ in range(max_steps):                 # hard cap: never loop forever
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=1024,
            tools=tools,
            messages=messages,
        )
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason != "tool_use":         # final answer
            return next(b.text for b in response.content if b.type == "text")

        results = []
        for block in response.content:
            if block.type == "tool_use":
                results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": run_tool(block.name, block.input),
                })
        messages.append({"role": "user", "content": results})

    return "gave up: hit max_steps"

④ Add an eval harness

Reliability is a number, not a vibe. Collect a handful of inputs where you know the right answer, write a checker for each, run the agent, and print a success rate.

starter · reliability eval
# Cases: (input, checker). The checker returns True if the output is correct.
cases = [
    ("Who wrote Pride and Prejudice? Give one surname.",
        lambda out: "austen" in out.lower()),
    ("What is the capital of Australia?",
        lambda out: "canberra" in out.lower()),
    ("In what year did the first Moon landing happen?",
        lambda out: "1969" in out),
]

passed = 0
for question, checker in cases:
    answer = run_agent(question)
    ok = checker(answer)
    passed += ok
    print(f"[{'PASS' if ok else 'FAIL'}] {question}  ->  {answer!r}")

print(f"\nreliability: {passed}/{len(cases)} = {passed / len(cases):.0%}")

What to hand in

Three things:

  • A repo — your tool definitions, the agent loop, and the eval harness, with a short README explaining the task.
  • A transcript or short demo — one run showing the agent reasoning, calling a tool, observing, and answering.
  • Your eval numbers — the success rate over your test cases, and a line on where it failed and why.
◆ Success criteria — you've passed when…

✓ Your agent uses a real tool inside a loop — not a single one-shot call.
✓ It handles at least one genuinely multi-step case (reason → act → observe → answer).
✓ You measured reliability — a real success rate over more than one input.
✓ The loop is bounded (a max-steps cap) so it can't run away.

▸ Stretch goal

Go one level up: add a second agent — a coordinator that decomposes the task and delegates to a specialist (5.7) — or expose your tool as an MCP server (5.6) so any MCP client can use it, not just your script. Then re-run your eval and compare the numbers.
