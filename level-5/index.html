Lesson 5.0

ML foundations in one lesson

Strip away the hype and machine learning is one idea: find parameters that minimize error. You don't write the rules — you write a model with tunable knobs, define how wrong it is, and let an optimizer turn the knobs until it's less wrong. Four moving parts:

  • Data — examples the model learns from. Often (input, target) pairs.
  • Model — a function with parameters (weights) that maps input → prediction.
  • Loss — a single number measuring how far predictions are from targets.
  • Gradient descent — nudge each parameter in the direction that lowers the loss, repeat.

The three classic problem shapes:

  • Supervised — labelled data; learn input → target (spam/not-spam, price prediction).
  • Unsupervised — no labels; find structure (clustering, dimensionality reduction).
  • Reinforcement — learn from rewards for actions in an environment (game-play, RLHF later in 5.4).

Here is the entire loop — fit a line to points with nothing but NumPy. Watch w and b walk toward the true values:

python · gradient descent
import numpy as np

# Model: y = w*x + b. We'll learn w and b. (True line: y = 2x + 1)
X = np.array([1, 2, 3, 4])
y = np.array([3, 5, 7, 9])

w, b = 0.0, 0.0        # parameters, start at zero
lr = 0.01              # learning rate: how big each step is

for step in range(2000):
    y_hat = w * X + b                 # forward pass: predict
    error = y_hat - y                 # how wrong, per point
    loss  = (error ** 2).mean()       # mean squared error

    grad_w = (2 * error * X).mean()   # d(loss)/d(w)
    grad_b = (2 * error).mean()       # d(loss)/d(b)

    w -= lr * grad_w                  # step downhill
    b -= lr * grad_b

print(round(w, 2), round(b, 2))       # -> 2.0 1.0
◆ Insight

Everything else in this level is this loop at scale. A large language model is the same idea — a model, a loss, gradient descent — with billions of parameters instead of two, and "predict the next token" as the target. The mechanics don't change; only the size does.

Lesson 5.1

Neural networks & deep learning

A neuron is just a weighted sum of its inputs plus a bias, passed through a non-linear activation (like ReLU: max(0, x)). Stack neurons into a layer, stack layers into a network. The forward pass runs data through the layers to a prediction; the loss says how wrong it was.

The learning happens via backpropagation: the chain rule computes, for every weight in the network, how much it contributed to the loss — its gradient. Then it's the same gradient descent from 5.0: nudge each weight downhill. That's the whole trick. "Deep" just means many layers.

Why did this suddenly work after decades of theory? Three things arrived together: data (the internet), compute (GPUs), and scale (bigger models keep getting better). None alone was enough.

The same network as before, now in PyTorch — note it's still forward → loss → backward → step:

python · pytorch
import torch
import torch.nn as nn

# 784 pixels in -> 128 hidden -> 10 class scores out
model = nn.Sequential(
    nn.Linear(784, 128),   # weights + biases (the learned parameters)
    nn.ReLU(),             # activation: adds non-linearity
    nn.Linear(128, 10),
)

loss_fn   = nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

for x, target in training_batches:   # x: images, target: labels
    pred = model(x)                  # forward pass
    loss = loss_fn(pred, target)

    optimizer.zero_grad()
    loss.backward()                  # backprop: gradient for every weight
    optimizer.step()                 # gradient descent: update the weights
● Note

You rarely write loss.backward()'s math by hand — frameworks like PyTorch build a computation graph and differentiate it automatically (autograd). Your job is to design the model, pick the loss, and feed it good data. The optimizer does the calculus.

Lesson 5.2

Embeddings & vector space

An embedding turns a piece of content — a word, a sentence, an image — into a list of numbers (a vector) such that meaning becomes geometry. Things that mean similar things land close together in the vector space; unrelated things land far apart. This is the representation modern AI thinks in.

"Close" is measured with cosine similarity — the angle between two vectors, from 1 (identical direction) to −1 (opposite). Once meaning is geometry, three big capabilities fall out for free:

  • Semantic search — embed the query, return the nearest documents (matches meaning, not keywords).
  • Clustering — group vectors that huddle together to find topics or duplicates.
  • RAG (retrieval-augmented generation) — embed a knowledge base, retrieve the passages nearest the question, and hand them to an LLM as context. This is how you ground a model in your data.

See it directly — two ways of saying the same thing score high; an unrelated sentence scores low:

python · embeddings
from sentence_transformers import SentenceTransformer
import numpy as np

model = SentenceTransformer("all-MiniLM-L6-v2")   # text -> 384-dim vectors

docs = [
    "the cat sat on the mat",
    "a feline rested on the rug",
    "quarterly revenue rose 12 percent",
]
vecs = model.encode(docs)                          # shape (3, 384)

def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

print(round(cosine(vecs[0], vecs[1]), 2))   # high  (~0.6) same meaning, different words
print(round(cosine(vecs[0], vecs[2]), 2))   # low   (~0.1) unrelated
◆ Insight

A vector database (FAISS, pgvector, Pinecone) is just this at scale: store millions of embeddings, find the nearest k in milliseconds. RAG isn't magic — it's a nearest-neighbour lookup feeding a prompt. Understanding that demystifies most "chat with your docs" products.

Lesson 5.3

Fine-tuning & adaptation

Fine-tuning continues training a pretrained model on your own data so it internalizes a style, format, or domain. Two flavours:

  • Full fine-tuning — update every weight. Powerful, but expensive: you need a lot of data, GPUs, and you get a whole new multi-gigabyte model to host.
  • Parameter-efficient fine-tuning (PEFT) — freeze the base model and train tiny add-on matrices. LoRA is the popular one: it injects small low-rank updates into a few layers, training well under 1% of the parameters. Cheap, fast, and you can swap adapters like plugins.
python · lora / peft
from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3.1-8B")

lora = LoraConfig(
    r=16,                                  # rank of the low-rank update
    lora_alpha=32,
    target_modules=["q_proj", "v_proj"],   # adapt the attention projections
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)

model = get_peft_model(base, lora)
model.print_trainable_parameters()         # e.g. ~0.06% of params trainable

But the most important skill here is knowing when not to. The decision order, cheapest and fastest first:

  1. Prompt first. Better instructions, examples, and a system prompt solve a surprising amount — with zero training and instant iteration.
  2. RAG next. If the problem is "the model doesn't know my facts," retrieve them (5.2). Facts change; retraining to teach facts is the wrong tool.
  3. Fine-tune last. Reach for it when you need a consistent behaviour, tone, or format that prompting can't hold, and you have quality labelled data. It teaches skills and style, not fresh facts.
▲ Careful

Fine-tuning is the answer people reach for too early. It bakes knowledge in at training time, so it goes stale, costs money to redo, and can't cite a source. If your real need is "know the latest" or "know my private data," that's RAG. Prove prompting and RAG can't do it before you spin up a training run.

Lesson 5.4

Alignment: shaping helpful, honest, harmless models

A model that has only been trained to predict the next token is a brilliant autocomplete — not an assistant. It has no notion of being helpful, telling the truth, or refusing harm. Alignment is the set of techniques that shape a raw pretrained model into something that behaves the way we want.

The pipeline, conceptually:

the alignment pipeline
1. PRETRAIN     predict-next-token on a huge corpus
                -> a model that *completes* text

2. SFT          supervised fine-tune on curated (prompt, good-answer) pairs
                -> a model that *follows instructions*

3. PREFERENCES  collect comparisons: for one prompt, "answer A is better than B"

4a. RLHF        train a reward model on those preferences, then use RL (PPO)
                to push the policy toward answers the reward model scores high

4b. DPO         Direct Preference Optimization: skip the separate reward model,
                optimize the policy directly on the preference pairs
                (simpler and widely used)

Constitutional AI (Anthropic's approach): the model critiques and revises its
own answers against a written set of principles (a "constitution"). Much of the
feedback is AI-generated (RLAIF) rather than needing a human label per example.

The through-line: after pretraining teaches raw capability and SFT teaches instruction-following, we steer values with human (or AI) preferences. RLHF learns a reward model from those preferences and optimizes against it. DPO reaches a similar place without the separate reward model, straight from the preference pairs. Constitutional AI — Anthropic's method — has the model use an explicit set of written principles to critique and improve its own responses, so the preference signal can be generated at scale by the model itself.

● Note

Alignment is why a well-trained model refuses harmful requests, hedges when unsure, and doesn't just parrot the most likely continuation. It's a genuinely open research area — "helpful, honest, and harmless" is easy to say and hard to fully specify. Treat any single technique as a current best effort, not a solved problem.

Lesson 5.5

Agentic AI foundations

A chatbot answers a message. An agent pursues a goal — it can take actions, observe the results, and decide what to do next, in a loop, until the goal is met. Four capabilities separate the two:

  • Perception — read the goal, the conversation, and tool results (its view of the world).
  • Planning — reason about what to do next.
  • Tool use — call functions, APIs, code, or other agents to actually do things.
  • Memory — carry state across steps (the running transcript, a scratchpad, or a store).

Tying them together is a control loop. The dominant pattern is ReAct — reason → act → observe → repeat: the model reasons about the next step, acts by calling a tool, observes the result, and loops with that new information until it's done.

Here's a real ReAct loop against the Claude API. The model reasons and asks for a tool; you run it and feed the result back; it continues until it stops requesting tools:

python · claude agent loop
import anthropic

client = anthropic.Anthropic()

# REASON (Claude thinks) -> ACT (asks for a tool)
# -> OBSERVE (you run it) -> repeat, until it stops asking for tools.
messages = [{
    "role": "user",
    "content": "What's the weather in Paris, and should I take an umbrella?",
}]

while True:
    response = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        tools=tools,                 # tool definitions from 5.6
        messages=messages,
    )
    messages.append({"role": "assistant", "content": response.content})

    if response.stop_reason != "tool_use":
        break                        # Claude gave a final answer -> done

    # ACT + OBSERVE: run each requested tool, feed results back in one turn
    results = []
    for block in response.content:
        if block.type == "tool_use":
            output = run_tool(block.name, block.input)   # your code
            results.append({
                "type": "tool_result",
                "tool_use_id": block.id,
                "content": output,
            })
    messages.append({"role": "user", "content": results})

print(next(b.text for b in response.content if b.type == "text"))
◆ Insight

Notice the loop is your code, not the model's. The LLM is a reasoning engine that emits "I want to call tool X with these arguments"; your harness decides what tools exist, executes them, and controls the loop — including when to stop. That separation is the whole architecture of an agent.

Lesson 5.6

Tool use & the Model Context Protocol

Tool use (function calling) is how a model reaches beyond text. You describe your tools as JSON Schema; the model, when it decides a tool would help, returns a structured request for it. It never runs anything — it just asks, and your code decides whether and how to execute.

Each tool is {name, description, input_schema}. Send them with your message; a good description is what teaches the model when to reach for the tool:

python · defining a tool
import anthropic

client = anthropic.Anthropic()

# 1. Describe the tool: name, description, and a JSON Schema for its input.
tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a city. "
                   "Use this whenever the user asks about weather.",
    "input_schema": {
        "type": "object",
        "properties": {
            "city": {"type": "string", "description": "City name, e.g. Paris"},
        },
        "required": ["city"],
    },
}]

def run_tool(name, args):
    if name == "get_weather":
        # call a real weather API here; hard-coded for the example
        return f"18C, light rain in {args['city']}"
    raise ValueError(name)

# 2. One turn. If Claude wants the tool, stop_reason == "tool_use".
response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[{"role": "user", "content": "What's the weather in Paris?"}],
)

for block in response.content:
    if block.type == "tool_use":
        print(block.name, block.input)   # -> get_weather {'city': 'Paris'}

Wrap that in the loop from 5.5 and you have a working agent. But notice the friction: every tool is a bespoke integration you write and maintain. That's the problem the Model Context Protocol (MCP) solves.

MCP is an open standard (introduced by Anthropic) for connecting AI applications to tools, data, and services. Instead of hand-coding each integration per app, a capability is exposed once as an MCP server; any MCP-compatible client — Claude, an IDE, your own agent — can discover and call it. Think of it as a universal port for agents: write the connector once, plug in anywhere.

json · mcp server config
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/data"]
    },
    "git": {
      "command": "uvx",
      "args": ["mcp-server-git", "--repository", "/data/repo"]
    }
  }
}
▸ Try it

Point an MCP-aware client at a prebuilt server (filesystem, GitHub, a database) and ask the model to use it — no glue code. Then read the spec at modelcontextprotocol.io ↗ and picture wrapping one of your internal services as a server. That's the leap from a one-off tool to a reusable capability.

Lesson 5.7

Multi-agent systems & the frontier

One agent has one context window and one line of attention. For big, decomposable jobs, a multi-agent system can do better: several specialized agents, each with its own role, tools, and prompt, coordinated toward a shared goal.

  • Roles — a researcher, a coder, a writer, a critic; each tuned for its job.
  • Hand-offs — one agent's output becomes another's input.
  • A coordinator — an orchestrator that decomposes the task, dispatches sub-tasks, and synthesizes the results.
python · coordinator pattern
# A coordinator delegates sub-tasks to specialists, then combines results.
def coordinator(task):
    plan = llm(f"Break this into independent sub-tasks:\n{task}")   # decompose

    results = []
    for subtask in plan.subtasks:
        agent = pick_specialist(subtask)     # "researcher" | "coder" | "writer"
        results.append(agent.run(subtask))   # each runs its own tool-use loop

    return llm(f"Combine into one final answer:\n{results}")        # synthesize

More autonomy means more ways to fail, so evaluation stops being optional. You can't eyeball reliability — you measure it: run the agent on a fixed set of inputs and score each result against a success criterion.

python · reliability eval
# Run the agent on labelled cases and compute a success rate.
cases  = load_test_cases()          # list of (input, checker) pairs
passed = 0

for case in cases:
    output = agent.run(case.input)
    if case.checker(output):        # did it meet the criterion?
        passed += 1

print(f"success rate: {passed}/{len(cases)} = {passed / len(cases):.0%}")

This is the frontier, and it demands care. Autonomous systems that take real actions can fail in compounding ways, so safety and oversight are engineering requirements, not afterthoughts: keep a human in the loop for consequential actions, scope each tool's permissions tightly, log every step for audit, and set hard limits (max iterations, cost caps) so a loop can't run away.

▲ Careful

Capability and control scale together. An agent that can send email, spend money, or run code can do so wrongly at machine speed. Build the guardrails before you widen the autonomy: confirmations on irreversible actions, least-privilege tools, and reliability numbers you actually trust. Responsible frontier work is measured, gated, and observable — not "ship it and hope."
