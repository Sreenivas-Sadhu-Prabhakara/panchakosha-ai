Lesson 2.0

From chatting to systems

In Level 1 you got things done one chat at a time. That's a skill worth keeping — but at work the same requests come back every week: the same status update, the same kind of summary, the same tone of reply. Re-typing the context each time is where the effort leaks away.

The mindset shift of Level 2 is simple: stop re-explaining, start reusing. Anything you do more than twice becomes a small, saved setup — a stored instruction block, a prompt you keep in a note, an automation that fires on its own. You move from operating the assistant by hand to building the assistant a workshop.

A quick way to spot what's worth turning into a system:

  • Recurring — you'll do it again next week.
  • Context-heavy — you keep pasting the same background.
  • Shape-stable — the output should look the same each time.

Run this audit prompt against your own week to find your first candidate:

audit
Here's how I spend a typical work week: [paste 6-8 recurring tasks].
For each task, tell me: is it a good fit for a reusable AI setup (saved
instructions, a template, or an automation)? Score each 1-5 for time saved
and give a one-line reason. Then recommend which ONE to systematise first.
◆ Insight

Treat your prompts like assets, not keystrokes. A prompt you refine once and save is worth more than a hundred clever ones you type and lose. The rest of this level is just ways to save and connect them.

Lesson 2.1

Advanced prompting

Level 1's C·E·F·T gets you a good answer. These five techniques get you a reliable one — the same quality across different inputs. Reach for them when the stakes or the volume rise.

  • Personas / roles — "You are a careful contracts lawyer" primes the right vocabulary and caution.
  • Few-shot examples — show 2–3 input→output pairs and the model matches your pattern far better than any description.
  • Chain-of-thought — "think step by step before you answer" trades a little speed for a lot of accuracy on reasoning tasks.
  • Structured output — ask for JSON, a table, or a fixed schema so the result is machine- and eye-readable.
  • Delimiters — wrap pasted material in clear markers so the model never confuses your data with your instructions.

Persona + few-shot, working together to classify support messages:

persona + few-shot
You are a support triage assistant. Label each message as BUG, BILLING,
or FEATURE. Match the style of these examples:

"App crashes when I upload a photo" -> BUG
"I was charged twice this month" -> BILLING
"Can you add dark mode?" -> FEATURE

Now label: "The export button does nothing when I click it"

Chain-of-thought, for anything with steps or trade-offs:

chain-of-thought
Two vendors quoted us. A: $4,200 upfront. B: $290/month, cancel anytime.
Think step by step about break-even and risk before you recommend one.
Show your reasoning, then give a one-line recommendation.

Structured output, so the answer drops straight into a spreadsheet or script:

structured output → JSON
Extract the action items from the notes below. Return ONLY valid JSON,
no prose, matching this schema:

{ "items": [ { "task": string, "owner": string, "due": string|null } ] }

Notes:
"""
Priya to send the revised deck by Thu. Sam will book the venue.
We still need someone to confirm catering numbers.
"""

Note the """ delimiters above — they fence off your data so it can't be mistaken for an instruction. XML-style tags like <notes>…</notes> work just as well and are worth using whenever a paste is long.

● Note

These stack. A production-grade prompt often is a persona + two examples + "think step by step" + a JSON schema + delimiters. Add one technique at a time and stop when the output is dependable — no need to use all five.

▸ Try it

Take a prompt you already reuse and add just the structured-output rule ("return a markdown table with columns X, Y, Z"). Run it on three different inputs. Notice how the shape holds steady even when the content varies.

Lesson 2.2

Projects, custom assistants & system prompts

The single highest-leverage move in this whole level: write your context once and reuse it forever. Every major tool now offers a container for this — a place where your instructions and reference files live so you never paste them again:

  • Claude Projects — a workspace with custom instructions plus knowledge files the assistant always sees.
  • Custom GPTs (ChatGPT) — a named assistant with a saved instruction block and optional files.
  • Gemini Gems — Google's version of the same idea, inside Gemini.

The heart of each is the system prompt (sometimes called "instructions"): a standing brief that shapes every reply. A good one names the role, the audience, the format, the boundaries, and how to handle uncertainty. Here's a reusable template for a "Weekly Update" Project:

system prompt · reusable
ROLE: You are my writing partner for weekly team updates.

AUDIENCE: A busy engineering director who skims. Lead with outcomes.

FORMAT: Markdown. Three headers — Shipped, In progress, Risks.
Max 5 bullets each. Every bullet starts with a verb. No filler.

VOICE: Plain, confident, honest. Never inflate. If something slipped,
say so in one calm line.

RULES:
- If I give you raw notes, turn them into the update above.
- If a detail is missing (a date, an owner), ask ONE question, then wait.
- Never invent metrics. If I didn't give a number, don't state one.

Paste that once into a Project's instructions, drop in your rough notes each week, and the output is consistent no matter who's tired or rushed. The same block works as a custom GPT or a Gem — the idea is portable even if the buttons differ.

◆ Insight

The "RULES" section is what separates a toy from a tool. Explicitly telling the assistant what to do when information is missing ("ask one question, then wait" · "never invent metrics") is what makes a reusable setup safe to trust.

▸ Try it

Create one Project (or custom GPT / Gem) for the task you found in Lesson 2.0. Give it a system prompt, add one reference file if you have one, and run it twice this week. You'll feel the difference on the second run.

Lesson 2.3

AI for coding

Coding assistants are no longer just for engineers. Whether you write software daily or have never opened a terminal, the same tools now help you build a script, a formula, a small web page, or a fix — by describing what you want in plain language.

  • Claude Code — an agentic assistant that reads and edits real files in your project from the command line.
  • Cursor — a full code editor with AI woven throughout; great for larger projects.
  • GitHub Copilot — autocomplete-style suggestions and chat inside popular editors.

The reliable pattern is describe → generate → review → refine. Be specific about inputs, outputs, and constraints — the same C·E·F·T discipline, aimed at code:

describe → generate
Write a Python script that reads sales.csv (columns: date, region, amount),
sums the amount per region, and writes region_totals.csv sorted high to low.
Handle missing amounts as 0. Add short comments. Keep it beginner-readable.

Generating is the easy 20%. Reviewing is the 80% that keeps you safe — never ship code you can't at least skim and explain. Ask the assistant to review its own work with you:

review
Explain this script line by line as if to a non-programmer. Then list what
could go wrong (bad input, empty file, wrong column names) and how you'd
guard against each. Don't rewrite it yet — just the review.

Here's a preview, not a task — you don't need to run this now. When you do write code later in this path, the API calls follow one consistent shape (they run in Python with an API key, which you'll set up in Level 4):

python · anthropic SDK
message = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    messages=[
        {"role": "user", "content": "Summarise this CSV in 3 bullets: ..."}
    ],
)
print(message.content[0].text)
▲ Careful

Beware "vibe coding" — accepting generated code you don't understand because it looks right and seems to run. It may hide security holes, wrong edge-case handling, or silent data loss. Rule: if you can't explain what a block does, don't ship it until you can — or until someone who can has reviewed it.

Lesson 2.4

Data & documents at work

Two of the biggest time sinks in office work — wrangling spreadsheets and reading long documents — are where assistants shine, provided you keep them honest with structure and verification.

  • Spreadsheets & CSVs — upload a file and ask for trends, pivots, outliers, or the exact formula to compute something.
  • Long PDFs — pull the parts that matter (dates, obligations, figures) into a clean table.
  • Comparing documents — diff two versions of a contract or policy and get the changes that actually matter.

Turn a 40-page PDF into a table you can act on:

extract → table
I've attached a vendor contract. Extract every obligation into a table with
columns: Clause #, Who is responsible, What they must do, Deadline, Penalty
if missed. Quote the clause number exactly. If a field isn't stated, write
"not specified" — never guess.

Compare two versions and surface only the meaningful deltas:

compare
Here are v1 and v2 of our refund policy. List only the substantive changes
(rights, timelines, money). Ignore wording/formatting tweaks. For each, show
the old text, the new text, and why it matters to a customer.
▲ Careful

Assistants can misread a figure or a date in a dense document — and state it just as confidently as a correct one. For anything that drives a decision, spot-check the extracted numbers against the source. Structure ("quote the clause number", "not specified if unstated") makes errors easier to catch, but doesn't remove your obligation to verify.

Lesson 2.5

Automation & connecting tools

So far you've been the one pressing "send". The next step is letting the assistant run without you in the loop for the routine parts. Two friendly on-ramps need no code:

  • Zapier and Make — visual builders that chain apps together: "when X happens, do Y". Many now have an AI step you can drop into the chain.
  • Scheduled prompts — some assistants can run a saved prompt on a timer (e.g. every Monday 8am) and deliver the result to you.

You describe an automation in plain English, and the builder wires it up:

zapier-style automation
TRIGGER: A new row is added to my "Inbound leads" Google Sheet.
ACTION 1 (AI): Draft a warm 3-sentence intro email to the lead, using their
               name and company from the row. Professional, no hard sell.
ACTION 2: Create a Gmail DRAFT (not send) addressed to the lead.
ACTION 3: Post the lead's name to our #sales Slack channel.
GUARDRAIL: Never auto-send. I review and send every draft myself.

Underneath the friendlier tools, there's an emerging standard for letting assistants safely reach your apps and data called MCP (Model Context Protocol) — think of it as "a standard plug that gives an assistant access to specific tools and data". You don't need it yet; it's the backbone of Level 5, where we take it apart properly.

● Note

Start every automation in draft / dry-run mode — generate, but let a human approve. Only remove the human once you've watched it behave correctly on real inputs for a while. Automating a bad prompt just makes mistakes faster.

Lesson 2.6

Multi-step work & light agents

Big tasks fail when you hand them over as one giant demand. They succeed when you make the assistant plan first, then execute — the same way a good colleague would sketch an approach before diving in.

The move is to split "do it" into two turns: agree the plan, then run it step by step.

plan first
Goal: produce a one-page competitor comparison for 3 rivals.
Before doing any of it, give me a numbered PLAN: what steps, in what order,
what you'll need from me, and where you might get things wrong.
Do NOT start until I reply "go".

Once you approve the plan, ask it to work the checklist and mark progress:

execute the checklist
Go. Work through the plan one step at a time. After each step, show what you
produced and tick it off. Pause and ask me if a step needs a decision or
information you don't have. Don't skip ahead.

When an assistant plans, acts, checks its own result, and adjusts — looping until the goal is met — it's behaving like an agent. Right now you're the loop: you approve each step. True agents close that loop themselves, and that's exactly what Level 5 is about. What you're learning here is the manual version of the same idea.

◆ Insight

"Plan, then execute" is the cheapest reliability upgrade you can make. Reviewing a five-line plan takes ten seconds and catches wrong assumptions before the assistant spends effort building the wrong thing.

Lesson 2.7

Quality, verification & responsible use at work

Using AI casually is forgiving. Using it at work — where output touches customers, colleagues, money, or your name — needs a little discipline. Three habits carry most of the weight.

1. Keep a small test set. The lightweight version of an "eval": save 5–10 tricky, real inputs your setup should handle. Whenever you change a prompt or switch models, run them and eyeball the results. It's the difference between "seems fine" and "still works".

a tiny eval set
Here are 6 test cases for my summariser (input → what a good output must
include). Run my current prompt on each and tell me, per case: PASS or FAIL
and why. Flag any where it invented, omitted, or softened a key fact.

1. [tricky input] -> must mention the missed deadline
2. [empty-ish input] -> must say "not enough information"
3. ...

2. Protect PII and company data. Before pasting, ask who can see this and whether it's used for training. Prefer enterprise/team plans with data protections for anything sensitive; redact names, IDs, and secrets you don't need the answer to include.

3. Disclose, and know when not to use it. Say when AI meaningfully drafted something if honesty calls for it. And skip AI entirely when a task demands guaranteed accuracy you can't verify, genuine confidentiality, or accountable human judgement (legal filings, medical decisions, anything where "the AI said so" isn't an acceptable answer).

▲ Careful

The workplace version of Level 1's golden test: "If this exact input leaked or this exact output shipped unread, would I be fine?" If not, redact the input, or verify the output, before it leaves your hands. You are accountable for what the assistant produces under your name.

Lesson 2.8

Your AI toolkit — apps beyond the chatbot

A chatbot is your general-purpose engine — brilliant for drafting, thinking, and analysis. But for specific jobs — a slide deck, a podcast edit, a meeting recap, a voiceover — a purpose-built AI app will beat a raw prompt every time. It already knows the format, handles the media, and exports the finished thing.

Here's a working directory of the tools professionals actually reach for in 2026, grouped by the job to be done. You don't need all of them — skim for the jobs you do weekly and try one.

◆ Insight

Learn the job first, then the tool. Apps come and go, but "turn notes into a deck" or "cut the filler from a recording" are permanent needs. Pick tools by the job they win, and swapping one out later costs you nothing.

Judge any new AI tool in 60 seconds

  • Job fit — does it do one real job noticeably better than your chatbot? If not, skip it.
  • Your sample — try it on your actual content, not the demo. Quality on your material is the only test that counts.
  • Data terms — what happens to what you upload? Is it used for training? Can you turn that off? (This matters double at work — see 2.7.)
  • Fits your stack — does it export to, or connect with, the tools you already use? An island tool creates more work than it saves.

Two workflows to try today

A deck from rough notes, in one Gamma prompt:

gamma · prompt
Turn these notes into a 10-slide deck: [paste your notes].
One idea per slide, a metrics slide near the end, and a closing slide with a
clear ask. Tone: confident and concrete, no filler. Suggest a simple visual
for each slide.

A clean clip from a raw recording, in Descript, end to end:

  1. Record or import your video/audio — it auto-transcribes.
  2. Delete filler words ("um", "like") across the whole file in one click.
  3. Edit the transcript like a doc — deleting text deletes the media.
  4. Studio Sound to clean up echo and background noise.
  5. Export a short, captioned clip ready for social.
● Note

This landscape moves fast — tools launch, merge, and change pricing constantly, and most have a free tier worth testing first. Treat this as a starting map, not gospel: run the 60-second test above before you commit real work (or budget) to any of them.
