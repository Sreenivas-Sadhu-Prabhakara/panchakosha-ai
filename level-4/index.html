Lesson 4.0

The shape of an AI app

Almost every AI product has the same three-part shape: a client (a browser, a phone app, a CLI), your backend, and the model API. The one rule that decides the whole architecture: your API key lives on the backend, never in the client. Anything shipped to a user's device can be read by that user.

So the flow is always: the client sends a request to your server, your server calls the model with your secret key, and you return just the answer.

the request loop
 Browser / app          Your backend             Anthropic API
 (no API key)           (holds the key)          (the model)
      │                       │                        │
      │  POST /ask  ────────▶ │                        │
      │                       │  messages.create ────▶ │
      │                       │                        │  (Claude thinks)
      │                       │  ◀──── response ────── │
      │  ◀──── JSON answer ── │                        │

Here's that backend as a tiny, real service. It's the seed every later lesson grows from:

server.py · FastAPI
import anthropic
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()
client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY from the environment

class Ask(BaseModel):
    question: str

@app.post("/ask")
def ask(body: Ask):
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": body.question}],
    )
    return {"answer": msg.content[0].text}

# run it:  pip install anthropic fastapi uvicorn  (the SDK + web deps)
#          uvicorn server:app --reload
◆ Insight

Everything in this level is a variation on that one endpoint. RAG adds a retrieval step before the model call; guardrails add checks around it; evals test it; deployment ships it. Get the loop clear in your head and the rest is plumbing.

Lesson 4.1

Your first API call

The whole API is one endpoint — messages.create. You pass a model, a max_tokens cap, an optional system prompt, and a list of messages. You get back content blocks; the text is in content[0].text. Set ANTHROPIC_API_KEY in your environment first (grab one from the console).

python
import anthropic

client = anthropic.Anthropic()   # reads ANTHROPIC_API_KEY

msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    system="You are a helpful assistant.",
    messages=[{"role": "user", "content": "Hello, Claude"}],
)
print(msg.content[0].text)

The same call in JavaScript / TypeScript, using the official SDK:

javascript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();   // reads ANTHROPIC_API_KEY

const msg = await client.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 1024,
  system: "You are a helpful assistant.",
  messages: [{ role: "user", content: "Hello, Claude" }],
});
console.log(msg.content[0].text);

Networks fail and rate limits happen. The SDK already retries 429 and 5xx errors with exponential backoff (twice by default), so you only add a try/except for the errors you want to surface:

python · errors
import anthropic

client = anthropic.Anthropic()

try:
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": "Hello, Claude"}],
    )
    print(msg.content[0].text)
except anthropic.RateLimitError:
    print("Rate limited — the SDK already retried; back off and try again later.")
except anthropic.APIStatusError as e:
    print(f"API error {e.status_code}: {e.message}")
▲ Careful

Never put your API key in frontend code. No NEXT_PUBLIC_ANTHROPIC_API_KEY, no key in a mobile bundle, no key in client-side JavaScript. It will be extracted and abused. Always call the model from a backend you control (Lesson 4.0).

Lesson 4.2

Prompting in code

Three moves cover most app prompting: a system prompt to set the role and rules, multi-turn messages to hold a conversation, and structured output when your code needs to consume the result.

The API is stateless — you send the whole history every time, alternating user and assistant:

python · multi-turn
messages = [
    {"role": "user",      "content": "My name is Ada."},
    {"role": "assistant", "content": "Nice to meet you, Ada!"},
    {"role": "user",      "content": "What's my name?"},
]
msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=256,
    system="You are a concise assistant.",
    messages=messages,
)
print(msg.content[0].text)   # "Your name is Ada."

When your program needs to use the answer, ask for JSON and validate it — never trust the shape blindly:

python · structured output
import json

msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system="Extract the booking. Reply with ONLY a JSON object, no prose.",
    messages=[{"role": "user",
        "content": "Book a table for Ada, 4 people, Friday 7pm."}],
)

data = json.loads(msg.content[0].text)   # raises if the model returned junk
assert "party_size" in data              # validate before you trust it
print(data)

And when the model needs to do something — look up weather, query a DB — you give it tools. You pass a tools array and watch for stop_reason == "tool_use":

python · tool use
tools = [{
    "name": "get_weather",
    "description": "Get the current weather for a city.",
    "input_schema": {
        "type": "object",
        "properties": {"city": {"type": "string"}},
        "required": ["city"],
    },
}]

msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    tools=tools,
    messages=[{"role": "user", "content": "What's the weather in Pune?"}],
)

if msg.stop_reason == "tool_use":
    call = next(b for b in msg.content if b.type == "tool_use")
    print(call.name, call.input)   # get_weather {'city': 'Pune'}
    # → run your real get_weather(**call.input), then send the result
    #   back as a tool_result message and call the API again.
● Note

Asking for JSON works, but the model can still stray. For guaranteed-valid output, the API supports structured outputs (an output_config.format JSON schema) and strict tool use, which constrain the response to your schema. Reach for those once your parser starts throwing.

Lesson 4.3

Build a RAG pipeline

A model only knows what's in its training data and your prompt. Retrieval-Augmented Generation (RAG) lets it answer from your documents by fetching the relevant bits and pasting them into the prompt. Five steps:

  1. Chunk your documents into small passages.
  2. Embed each chunk into a vector (a list of numbers that captures meaning).
  3. Store the vectors in a vector database.
  4. Retrieve the top-k chunks most similar to the question.
  5. Augment the prompt with those chunks and ask Claude.

Here's the whole thing, small enough to read in one sitting (uses a local embedding model so it runs with no extra keys):

python · rag.py
import anthropic, numpy as np
from sentence_transformers import SentenceTransformer

client = anthropic.Anthropic()
embed = SentenceTransformer("all-MiniLM-L6-v2")   # local embedding model

# 1. CHUNK — your docs, split into small passages
docs = [
    "Refunds are issued within 5 business days.",
    "Our office is open Monday to Friday, 9am to 6pm.",
    "Free shipping applies on orders over 2000 rupees.",
]

# 2. EMBED + 3. STORE — one vector per chunk (in prod: pgvector / Pinecone / FAISS)
doc_vecs = embed.encode(docs)

def answer(question: str) -> str:
    # 4. RETRIEVE — top-2 chunks by cosine similarity
    q = embed.encode([question])[0]
    sims = doc_vecs @ q / (np.linalg.norm(doc_vecs, axis=1) * np.linalg.norm(q))
    top = [docs[i] for i in sims.argsort()[-2:][::-1]]

    # 5. AUGMENT — put the retrieved context in the prompt
    context = "\n".join(f"- {c}" for c in top)
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        system="Answer ONLY from the context. If it's not there, say you don't know.",
        messages=[{"role": "user",
            "content": f"Context:\n{context}\n\nQuestion: {question}"}],
    )
    return msg.content[0].text

print(answer("How long do refunds take?"))
# → "Refunds are issued within 5 business days."
◆ Insight

The toy pieces have production homes. For embeddings, Anthropic recommends Voyage AI (a hosted embedding model) over a local one for quality. For storage, use a real vector DB — pgvector if you already run Postgres, Pinecone for a managed service, or FAISS for an in-process index. The five steps stay identical; only the chunk store and embedder change.

Lesson 4.4

Deploy on a hyperscaler

Your service runs on your laptop; now put it on a URL. The simplest path for a small API is to containerize it and ship to a serverless container platform like Google Cloud Run (or package it as an AWS Lambda). A minimal Dockerfile:

Dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt
COPY . .
# Cloud Run routes traffic to $PORT (defaults to 8080)
CMD ["sh", "-c", "uvicorn server:app --host 0.0.0.0 --port ${PORT:-8080}"]

Deploy it — and keep the key in Secret Manager, injected as an environment variable at runtime, never baked into the image:

deploy · Cloud Run
# one command builds, pushes and deploys from source
gcloud run deploy rag-api \
  --source . \
  --region asia-south1 \
  --set-secrets ANTHROPIC_API_KEY=anthropic-key:latest \
  --allow-unauthenticated

# → Service URL: https://rag-api-xxxxx.a.run.app  (your public backend)
● Note

If your org already lives on a hyperscaler, you can reach Claude through it instead of the first-party API — Amazon Bedrock (the AnthropicBedrockMantle client) or Google Vertex AI (the AnthropicVertex client). Same messages.create surface; the difference is auth (IAM / ADC) and, on Bedrock, an anthropic. prefix on model IDs.

Lesson 4.5

Evals & testing

"It worked when I tried it" is not a test. Because model output varies, you check behaviour against a golden set — inputs paired with what a good answer must contain — and run it every time you change a prompt or model.

python · eval loop
import anthropic
client = anthropic.Anthropic()

# golden set: an input + something a correct answer must contain
GOLDEN = [
    {"q": "How long do refunds take?", "must_include": "5 business days"},
    {"q": "When are you open?",        "must_include": "Monday to Friday"},
]

def run(q: str) -> str:            # your app under test
    msg = client.messages.create(
        model="claude-sonnet-4-6", max_tokens=256,
        messages=[{"role": "user", "content": q}])
    return msg.content[0].text

passed = 0
for case in GOLDEN:
    out = run(case["q"])
    ok = case["must_include"].lower() in out.lower()   # a simple assertion
    passed += ok
    print("PASS" if ok else "FAIL", "—", case["q"])
print(f"{passed}/{len(GOLDEN)} passed")

Substring checks only get you so far. For open-ended answers, use LLM-as-judge: a stronger model grades the output of your cheaper production model.

python · llm-as-judge
def judge(question: str, answer: str) -> str:
    msg = client.messages.create(
        model="claude-opus-4-8",       # a stronger model grades the cheaper one
        max_tokens=10,
        system="Reply with only PASS or FAIL.",
        messages=[{"role": "user",
            "content": f"Q: {question}\nA: {answer}\n"
                       "Is the answer correct and helpful?"}])
    return msg.content[0].text.strip()

print(judge("When are you open?", run("When are you open?")))
▸ Try it

Run your golden set against claude-haiku-4-5-20251001 and again against claude-sonnet-4-6. Compare pass rates and latency. That table — quality vs speed vs cost — is exactly how you pick a production model instead of guessing.

Lesson 4.6

Guardrails, safety & cost control

A live endpoint takes untrusted input and spends real money. Two things wrap every model call: validation (reject bad or out-of-scope requests before they reach the model) and cost control (don't pay to think about things you shouldn't).

python · guard function
BLOCK = ("password", "credit card", "ssn")

def guard(user_input: str) -> str | None:
    """Return an error message if the request should be refused, else None."""
    if len(user_input) > 4000:
        return "Message too long — please shorten it."
    if any(word in user_input.lower() for word in BLOCK):
        return "Please don't share secrets like passwords or card numbers."
    return None

def ask(user_input: str) -> str:
    err = guard(user_input)                 # input validation
    if err:
        return err
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,                     # token budget: caps output cost
        system="You only answer questions about our shipping policy. "
               "Politely decline anything else.",   # refuse out-of-scope
        messages=[{"role": "user", "content": user_input}],
    )
    return msg.content[0].text

The biggest single lever on cost is prompt caching: mark a large, unchanging chunk of context (a policy doc, a system prompt) as cacheable and repeat calls reuse it far more cheaply.

python · prompt caching
msg = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=512,
    system=[{
        "type": "text",
        "text": BIG_POLICY_DOCUMENT,             # long, stable context
        "cache_control": {"type": "ephemeral"},  # cache it
    }],
    messages=[{"role": "user", "content": "What's the return window?"}],
)
print(msg.usage.cache_read_input_tokens)   # > 0 means you hit the cache
◆ Insight

Cost is driven by tokens, split into input (what you send) and output (what the model writes) — and output tokens cost more than input. Three levers: send less (cache, trim history), write less (a tight max_tokens), and use a smaller model (Haiku < Sonnet < Opus) where quality allows. For current numbers see anthropic.com/pricing and the API docs.

Lesson 4.7

Observability & iterating in prod

Once real users are hitting your endpoint, you need to see what they send, what the model returns, and what it cost — so you can debug failures and improve. Log a trace id, the token usage, and latency on every call.

python · logging & tracing
import logging, uuid, time
logging.basicConfig(level=logging.INFO)
log = logging.getLogger("ai")

def ask(user_input: str) -> dict:
    trace_id = str(uuid.uuid4())
    t0 = time.time()
    msg = client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=512,
        messages=[{"role": "user", "content": user_input}],
    )
    answer = msg.content[0].text
    log.info("trace=%s ms=%d in_tok=%d out_tok=%d",
             trace_id, (time.time() - t0) * 1000,
             msg.usage.input_tokens, msg.usage.output_tokens)
    return {"trace_id": trace_id, "answer": answer}

Then capture user feedback against that trace id and close the loop — every thumbs-down is a candidate for tomorrow's eval set.

python · closing the loop
def record_feedback(trace_id: str, helpful: bool):
    db.save({"trace_id": trace_id, "helpful": helpful})
    # Failing traces become new golden cases (Lesson 4.5),
    # which drive the next round of prompt changes. That loop —
    # ship → log → collect feedback → add evals → improve — is
    # what turns a demo into a product that gets better over time.
● Note

Store prompts and responses somewhere you can query them, but treat them as sensitive: redact secrets and personal data before logging, and check your privacy obligations before persisting user content. Observability should never become a second copy of data you promised to protect.
